# Release Procedure

Maintainer guide for publishing `ctx` releases.

---

## Required GitHub Secrets

Configure these in **Settings → Secrets and variables → Actions** for the
`CodeWithBryan/context` repository.

| Secret name | What it is | How to get it |
|---|---|---|
| `APPLE_DEVELOPER_ID_P12_BASE64` | Base64-encoded Developer ID Application `.p12` certificate | See below |
| `APPLE_DEVELOPER_ID_P12_PASSWORD` | Password protecting the `.p12` file | Set when exporting from Keychain |
| `APPLE_ID` | Apple ID email used for notarization | Your Apple developer account email |
| `APPLE_APP_SPECIFIC_PASSWORD` | App-specific password for `notarytool` | [appleid.apple.com](https://appleid.apple.com) → **App-Specific Passwords** → Generate |
| `APPLE_TEAM_ID` | 10-character alphanumeric team identifier | [developer.apple.com/account](https://developer.apple.com/account) → Membership → Team ID |

### Generating `APPLE_DEVELOPER_ID_P12_BASE64`

1. Open **Keychain Access** on your Mac.
2. Locate your **Developer ID Application** certificate (includes the private key).
3. Right-click → **Export** → choose **Personal Information Exchange (.p12)**.
4. Set a strong password — this becomes `APPLE_DEVELOPER_ID_P12_PASSWORD`.
5. Encode to base64:

   ```sh
   openssl base64 -in DeveloperID.p12 -out DeveloperID.p12.base64
   ```

6. Paste the entire contents of `DeveloperID.p12.base64` into the
   `APPLE_DEVELOPER_ID_P12_BASE64` secret. Delete the local `.p12.base64` file
   afterward.

> Do **not** commit `.p12` or `.p12.base64` files to the repository.

## Release Procedure

1. **Bump the version** in `Cargo.toml` (workspace root):

   ```toml
   [workspace.package]
   version = "0.1.1"   # was 0.1.0
   ```

2. **Commit** the version bump:

   ```sh
   git add Cargo.toml Cargo.lock
   git commit -m "chore: bump version to v0.1.1"
   ```

3. **Tag** the commit:

   ```sh
   git tag v0.1.1
   ```

4. **Push** the commit and tag together:

   ```sh
   git push origin main --tags
   ```

5. **Watch the Actions run** — go to the **Actions** tab in the repository and
   select the **Release** workflow. The typical run time is **10–15 minutes**,
   most of which is macOS notarization.

6. The **GitHub Release** is created automatically with:
   - Signed + notarized macOS ARM64 binary (Apple Silicon)
   - `SHA256SUMS` file
   - Auto-generated release notes from commit messages

   > Intel Mac and Linux builds are intentionally not published yet. Re-add
   > them to the matrix in `.github/workflows/release.yml` when needed.

## Testing Without a Real Release

Use the `workflow_dispatch` trigger to create a draft release without tagging:

1. Go to **Actions → Release → Run workflow**.
2. Click **Run workflow** (uses the `main` branch by default).
3. The workflow creates a **draft** release tagged `v<version>-rc-<run_id>`.
4. Download the binaries, install, and test locally.
5. **Delete the draft release** from the Releases page when done.
   (Draft releases do not affect `releases/latest` URLs.)

## Notes

- **Notarization timeout**: The `xcrun notarytool submit --wait --timeout 900`
  step waits up to 15 minutes. Apple's notarization service is typically faster
  (2–5 min), but can take longer during peak periods. If it times out, re-run
  only the `macos-sign` job from the Actions UI.

- **`releases/latest` URL**: GitHub's `/releases/latest/download/<asset>` always
  points to the most recent **non-draft, non-prerelease** release. Tags matching
  `v*-rc-*` are treated as prereleases if the release is created with
  `prerelease: true`; the workflow currently creates them as drafts instead, so
  they never pollute the `latest` pointer.

- **Re-signing after a binary change**: If the binary changes post-tag (e.g. you
  need to hotfix and re-tag), delete the old tag locally and remotely, create a
  new patch version, and repeat the procedure above. Do not overwrite published
  release assets.

- **Changelog**: Release notes are auto-generated by GitHub from commit messages
  (`generate_release_notes: true` in the workflow). For richer changelogs, add
  a `CHANGELOG.md` or configure `git-cliff` in a later phase.
